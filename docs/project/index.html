通用项目入口协议-Project Entry Protocol

定义一种跨语言、跨工具的项目声明、发现与执行标准。

// 1. 在任意目录放置标识文件声明逻辑项目
E:/go项目/Doc-war系列/效率工具/v1/myapp
├── src/
└── .project

// 2. 在系统的任何地方直接启动
$ project myapp

// 3. 结果：启动终端自动跳转至myapp目录。

运行机制

.project 本质上是一个“锚点”。
通过统一的 CLI 工具扫描你的工作目录，它在项目名与物理路径之间建立N-1关系的索引。

全局映射 不再需要手动 cd或频繁的切换资源管理器。无论项目在哪里，一条指令直达项目深处。
环境预设 在 .project 中定义 shell 类型和权限要求，确保每个项目都在正确的环境中运行。

为什么需要它？

在日常开发中，一台电脑可能有几十个项目散在各级文件夹。我们经常在不同的项目间，来回切换、启动终端、执行指令。这相当繁琐。

Node.js 后台服务:        D:/companyA/xxx/backend/user-service
Node.js 管理后台前端:    D:/companyA/xxx/frontend/admin-web
Node.js 支付沙盒项目:    D:/companyA/xxx/backend/payment/sandbox
Node.js 文档项目:        D:/companyA/xxx/docs/
React 实验项目:          E:/experiments/openai-test
Go 微服务(十几个):       D:/legacy/go/order-service
Python 数据处理脚本:     E:/scripts/data-pipeline
知识管理                 E:/markdowns/xxxx   (使用git管理了十几个typora项目)

在服务器部署时，即便是单项目，一个最简单的网站，也涉及nginx、前台服务、后台服务、前台前端、后台前端、数据库,由于前后端、配置通常分离，也非常繁琐。终端别名不能持久化，中心化脚本配置的导航器方案心智成本不佳

我们需要一个很干净的解决方案：足够轻量、足够灵活


.project 的目标是作为它们的统一入口。你只需要在项目根目录放置一个 .project 文件，即可通过简单的命令管理所有项目，然后通过项目名或序号快捷启动终端或打开资源管理器。

跨语言 适用于任何编程语言和框架，以及项目组合抽象。
无侵入 只是一个配置文件，不修改现有代码。
环境隔离 同一个文件夹支持组合放置来隔离配置，轻松切换开发和生产配置。

协议规范

项目发现

项目发现总是从用户配置的检索根目录开始，通过搜索.project 文件重建索引。

1、协议不定义固定的扫描深度、范围、忽略目标，由上层实现来选择体验策略。
2、以 . 开头的目录在主流语义中默认是非业务入口，明确要求不参与项目发现。
3、对于 node_modules 等非业务目录，强烈建议上层纳入忽略配置实现。

项目名与文件命名

所有以 .project 结尾的文件，均被视为合法的项目声明文件。

项目名称的确定遵循以下规则：
1、若协议文件显式具名，则使用文件名作为项目名；
2、若未具名，则默认使用 .project 所在目录名称。
3、允许同一个文件夹内部有多个`.project`文件，即支持同一个物理项目可以用不同的逻辑项目来打开。意味着你可以借助多个具名`.project`文件组合来实现策略隔离

文件配置格式

文件允许为空。如果配置，则必须是标准的 YAML 格式，且使用 UTF-8 编码。它可以被命名为 .project 或者 <名称>.project。

# 示例配置
commands:
  start: npm run dev
  build: ./scripts/build.sh
shell: wt
isAdmin: false

核心属性

-启动终端

属性名 类型 声明的语义
shell String 偏好的终端类型（如 wt, bash, powershell）。
isAdmin Boolean 是否需要管理员/Root 权限执行。
commands Map 自定义命令映射表，如 start, test 等。

-启动其他应用

属性名 类型 声明的语义
app String 用其他应用(比如typora、vscode)替代终端，来接管启动和接收路径上下文。

扩展属性

-扩展槽

属性名 类型 说明
extensions map{string,map} 上层扩展占位槽。

语义说明：
1、协议层永远不依赖其中任何字段，所有上层功能扩展必须放入此槽内。
2、extensions采用命名空间隔离，每个命名空间代表一个上层产品或能力单元，且更新操作时不应该破坏其他命名空间的原值。

官方实现

该协议目前已做了上层CLI工具实现：

Project (Go)

核心体验

1、轻量级的二进制工具，自动维护环境变量,启动一次任意执行

2、支持project快速罗列项目，使用序号精准启动

命令集

子命令具有最高优先级，项目名需避免与其冲突。
子命令 说明
project init 将当前目录声明为项目，交互式生成 .project 文件。
project search 扫描磁盘中的 .project 项目，并将发现结果写入本地缓存。
project root add|remove [path] 管理项目搜索的根目录集合；未传 path 时默认使用当前目录。
project default [shell] 设置或读取全局默认终端类型，用于统一项目启动环境，减少项目级配置。
project open <project> 打开指定项目所在目录的文件资源管理器。
project help 显示完整帮助信息与示例说明。
项目参数型命令表
命令结构 参数 类型 说明
project 无 Interactive 列出所有已缓存项目并进入交互模式，可通过[序号]精准选择并启动项目。
project <project> <project> Launch 启动指定项目。优先检查.project中的app字段，如有则启动app并传递路径上下文。否则默认启动终端。
project <project> <command> <project>
<command> 		Launch + Exec 启动指定项目，并在新终端中执行项目内配置的命令，如 dev、start、test 等。
